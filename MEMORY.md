## GitHub Pages 发布自检机制 (2026-03-04)

**决策 ID:** dec-20260304-123800

**规则:** 每次页面发布后必须执行自检 + 修复流程（最多 5 轮）。

**流程:**
1. 推送到 GitHub (master + gh-pages 分支)
2. 等待 30 秒
3. 自检：HTTP HEAD 请求检查页面是否 200
4. 如果失败 → 等待 30 秒 → 重试（最多 5 轮）
5. 5 轮后仍失败 → 自动修复（重新推送 gh-pages）
6. 修复后再自检 2 轮
7. 仍失败 → 通知用户手动检查

**脚本:** `scripts/mideast-monitor-v21.py`

**函数:**
- `verify_page_deployed(timestamp, max_retries=5)` - 自检
- `fix_deploy_if_needed(timestamp)` - 修复

## /new 命令跨平台会话管理机制 (2026-03-03)

**决策 ID:** dec-20260303-182400

**核心机制：**
- **dmScope: per-peer** - 会话按 peer_id 计算，忽略 channel
- **飞书/Telegram 共享同一 workspace** - 同一用户的不同平台共享 peer_id
- **/new 命令效果：** 同时清空两个平台的短期记忆

**执行流程：**
1. 保存当前 Session 到共享长期记忆 (MEMORY.md + 向量)
2. 归档每日快照 (memory/YYYY-MM-DD.md)
3. 销毁旧 Session 对象（短期记忆清空）
4. 创建全新 Session（新 agent-id + 清空 history）
5. 初始化新 workspace context（共享）

**可选参数：** `/new [model]` - 切换 LLM provider/model（如 `/new claude-3.5`）

**重要影响：** 在飞书执行 `/new` 会导致 Telegram 会话也重置，反之亦然。用户需知晓跨平台联动效应。

## 技术教训 (2026-03-03)

**决策 ID:** dec-20260303-155400

**事件：** 配置记忆搜索时服务挂掉

**原因：**
- 使用了 `openclaw configure --section model` 交互式命令
- 命令卡在交互界面，需要人选选项
- 尝试 piped 输入失败，会话卡住
- 最后重启 Gateway，导致服务中断 30-60 秒

**教训：**
- ❌ 禁止使用 `openclaw configure` 交互式命令
- ✅ 只用 `openclaw config set key value` 直接修改配置
- ✅ 配置修改后重启 Gateway 会导致服务中断，需提前告知

**影响：** Telegram 和 Feishu 中断约 1 分钟
